## Learnings

### Dither expansion origin shifts when magnetic button moves
**Context:** The footer CTA has a magnetic hover effect that translates the button toward the cursor. The dither dark-mode expansion was using `btn.getBoundingClientRect()` to calculate its radial origin, which meant the dither center moved with the magnetic offset, making it appear to radiate from the bottom third of the button instead of the center.
**Learning:** When calculating a fixed origin point for a radial animation tied to a magnetic/draggable element, use the static wrapper's bounding rect, not the moving element's. The wrapper stays put; the child moves.
**Why it matters:** Any time you pair a magnetic/parallax effect with a radial reveal, this same bug will appear.

### Drag cursors and clickable elements inside drag-scroll containers
**Context:** Case study cards had "Read the full story" links, but the drag cursor overlay covered them. The links inherited `cursor-none` from parent containers and the mousedown handler's `preventDefault()` blocked click events.
**Learning:** When adding clickable elements inside a drag-scroll container with a custom cursor: (1) hide the custom cursor on link mouseenter, restore on mouseleave, (2) set `cursor: pointer` on links to override inherited `cursor-none`, (3) guard the mousedown handler with `if (e.target.closest('a')) return` so links remain clickable.
**Why it matters:** Any scrollable gallery with overlay drag cursors will hit this if you add CTAs or links to the cards later.

### Mission section scroll calculation determines blank space perception
**Context:** In a previous session, the mission section (200vh with sticky content) showed a full blank screen before the animation triggered. It was fixed by changing scroll calc from `-rect.top` to `window.innerHeight - rect.top`. This session reduced height from 200vh to 150vh.
**Learning:** For scroll-driven sticky sections, `window.innerHeight - rect.top` starts the animation as the section enters the viewport from below, eliminating the dead scroll zone. Combined with reducing total section height, you can keep the sticky reveal effect without wasting scroll distance.
**Why it matters:** Scroll-driven animations are common in editorial sites. The calculation choice determines whether the user sees content or dead space.

### getBtnCenter() vs getWrapCenter() naming
**Context:** The function was called `getBtnCenter` but after the fix it uses `wrap.getBoundingClientRect()`. The name is now misleading.
**Learning:** (General) When you change what a function references internally, rename it. Left this one as-is for minimal diff, but it's a small debt.
**Why it matters:** Minor. Only matters if someone else reads this code.

## Skill & Automation Candidates

### deploy-and-verify
**What it would do:** After committing and deploying to Vercel, automatically fetch the live URL, take a screenshot, and present it for visual verification. Could also run a quick Lighthouse or contrast check.
**Why it's worth building:** Right now the workflow is commit > push > deploy > manually check the URL. Adding a verification screenshot step would catch visual regressions before sharing the link with a client.
**Trigger:** After any `vercel --prod` or `vercel` deploy command.
**Complexity:** Medium. Needs screenshot tooling (Puppeteer or similar) and some wait logic for Vercel build completion.

### interaction-audit
**What it would do:** Systematically check all interactive elements on a page for cursor conflicts, hover state issues, and click target overlap. Flag things like: custom cursors overlapping clickable elements, magnetic effects shifting animation origins, hover states that don't reset cleanly.
**Why it's worth building:** Three of the four visual fixes in this session were interaction conflicts that only surface when you actually use the page. These are easy to miss in code review.
**Trigger:** After building any page with custom cursors, magnetic effects, or scroll-driven animations.
**Complexity:** High. Would need to define a taxonomy of interaction patterns and their known conflicts. Probably better as a checklist than an automated tool.

No strong candidates from this session beyond these two. The conversion copy skills were already built in a prior session.

## LinkedIn Content

### Post 1: The bugs AI can't see

I spent an hour this week fixing four visual bugs on a homepage I'm building with Claude Code. Every single one was an interaction conflict that only shows up when you actually use the page.

A radial animation that was supposed to expand from the center of a button was expanding from the bottom third. A drag cursor was covering clickable links. A headline was wrapping to three lines when it needed to be two. A CTA was floating in space with no visual anchor to anything.

None of these showed up in the code. The code was fine. The logic was correct. The CSS was valid. But when you hover, scroll, resize, and click like a real person, all four were obvious within seconds.

This is the thing about designing in code with AI. The AI writes solid code. It handles structure, accessibility, conversion copy, responsive layouts. But it doesn't use the page. It doesn't hover over the button and notice the animation origin is off. It doesn't drag-scroll and realize the cursor is blocking the link it just added.

That's still the designer's job. Not the pixels. The experience of using the thing.

I fixed all four in about 20 minutes once I could see them. The seeing is the skill that matters.
